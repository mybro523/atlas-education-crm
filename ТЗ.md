# Техническое задание
## CRM-система для образовательного центра

> **Статус документа:** v0.2 (согласовано, дополнено)
> **Дата:** 2026-07-01
> **Рабочее название:** Atlas CRM (Education)

---

## 1. Общее описание

Веб-система управления образовательным центром (**сеть из 3 филиалов**) с ролевым доступом. Система охватывает
учебный процесс (курсы, группы, расписание, журнал, оценки, посещаемость), продажи (омниканальные
чаты с WhatsApp/Instagram), финансы (доходы/расходы, зарплаты, аналитика) и автоматические
уведомления (Telegram-боты для студентов и учителей, SMS родителям, SMS-рассылки).

**Основные бизнес-цели:**
- Единая платформа для всех процессов центра вместо разрозненных таблиц и мессенджеров.
- Прозрачность успеваемости для студентов и родителей.
- Автоматизация рутины: уведомления о пропусках, доступ к расписанию/оценкам через Telegram.
- Контроль финансов и зарплат в одном месте (только для основателя).

## 2. Технологический стек

## 4. Функциональные требования по ролям

### 4.1. Студент (личный кабинет)

- **Журнал:** список занятий по каждому предмету с оценками и отметками о пропусках (нб).
- **Успеваемость по предметам:** сводка по каждому предмету — средний балл, динамика, количество пропусков.
- **Расписание:** личное расписание занятий (день/неделя), с предметом, временем, аудиторией, учителем.
- **Профиль:** личные данные студента (ФИО, контакты, группа, родители) — только просмотр.
- Доступ к тем же данным через **Telegram-бота** (см. §6).

### 4.2. Учитель (личный кабинет)

- **Список групп**, которые ведёт учитель.
- **Журнал каждой группы** со списком студентов; возможность:
  - поставить **оценку**;
  - поставить **замечание**;
  - отметить **нб** (отсутствие / посещаемость).
- **Доступ к данным своих студентов:** телефоны студентов и родителей (для звонка).
- **Добавить студента в группу** (при приходе нового).
- **Удалить студента из группы** (при отчислении).
- Доступ к своему расписанию через **Telegram-бота**.

### 4.3. Менеджер продаж (кабинет)

- **Омниканальный инбокс:** входящие чаты из **WhatsApp** и **Instagram** приходят прямо в CRM.
- **Ответ из CRM:** менеджер отвечает клиенту внутри системы, сообщение уходит в исходный канал.
- **Доступ:** менеджер продаж (работа) + основатель (просмотр для контроля). Администратор — без доступа.
- (Карточки лидов, статусы сделок, привязка чата к студенту/группе — на этапе реализации модуля.)

### 4.4. Администратор (кабинет)

- **CRUD** учителей, студентов, групп, **курсов и типов курсов**.
- **Полный доступ** ко всем данным: студенты, учителя, группы (по всем 3 филиалам).
- **Расписание** всех групп и всех учителей (создание/редактирование).
- **Поиск студентов** по имени **и по месту работы родителей**.
- **SMS-рассылка** студентам и учителям (см. §7.3).
- **Всё, кроме раздела «Финансы»** — финансы недоступны.

### 4.5. Основатель (кабинет)

- **Всё, что доступно администратору**, плюс:
- **Финансовая аналитика по каждому из 3 филиалов + сводная** по всей сети.
- **Раздел «Финансы»:**
  - **Доходы / Расходы.**
  - **Оплаченные / Неоплаченные** (задолженности студентов). Оплата **помесячная**, при этом **месяц отсчитывается от даты вступления студента** (например, с 15-го по 15-е), а не по календарю; неоплаченный период = задолженность.
  - **Финансовая аналитика** с фильтрацией по времени (день/неделя/месяц/период). Валюта — **сомони (TJS)**.
  - **Зарплаты** учителей, менеджеров продаж, администраторов. Расчёт зарплаты учителя — **гибкий, по проведённым занятиям**: у каждого занятия/группы может быть своя ставка (например, одно занятие стоит 35 сомони, другое — 40 и т.д.). Итог за период = сумма по всем проведённым занятиям. У административных сотрудников возможен фиксированный оклад — ставки задаются в системе.

## 5. Модель данных (предварительно)

Основные сущности (детализируется на этапе проектирования БД):

- **User** — базовый пользователь (email/телефон, пароль-хэш, роль, язык интерфейса, telegram_chat_id, branch_id).
- **Branch** — филиал (название, адрес). Ссылку `branch_id` имеют студенты, учителя, группы, финансы. На текущем этапе — метка для фильтрации/аналитики (все видят все филиалы).
- **Student** — профиль студента (ФИО, дата рождения, контакты, родители[], группы[], курс, филиал, **дата вступления** — для расчёта платёжного месяца).
- **Parent** — родитель (ФИО, телефон, **место работы**) — **только для SMS-уведомлений, без входа в систему**. Место работы участвует в поиске студентов.
- **Teacher** — профиль учителя (ФИО, контакты, предметы[], филиал; ставки — см. LessonRate/Salary).
- **Employee** — администратор / менеджер продаж (для зарплат).
- **Course** — курс (название, **тип курса**, предмет, филиал, цена/месяц).
- **CourseType** — **справочник типов курсов** (Стандарт, ВИП, Интенсив, Детский, Для рабочих и любые новые). Управляется админом/основателем — **гибко, не хардкод**.
- **Group** — учебная группа (название, курс, предмет, учитель, студенты[], расписание, филиал).
- **Subject** — предмет.
- **Lesson** — конкретное занятие (группа, дата/время, тема, аудитория, ставка оплаты учителю за это занятие).
- **LessonRate** — ставка оплаты учителю за занятие/группу (**гибкая**: 35, 40 сомони и т.д.); основа расчёта зарплаты.
- **Schedule** — расписание (привязки Lesson к группам/учителям/аудиториям).
- **Grade** — оценка (студент, занятие/предмет, значение по **5-балльной шкале**: 2–5, дата, кто поставил).
- **Attendance** — посещаемость (студент, занятие, статус: присутствовал / нб / опоздал).
- **Remark** — замечание (студент, занятие, текст, автор).
- **Payment** — оплата (студент, группа/курс, **платёжный месяц от даты вступления**, сумма, дата, статус оплачено/не оплачено, филиал). Модель — помесячная.
- **FinanceRecord** — доход/расход (тип, сумма, категория, дата, описание, филиал). Валюта — сомони (TJS).
- **Salary** — зарплата сотрудника (сотрудник, период, база расчёта; для учителя = сумма ставок проведённых занятий, для админ-персонала = оклад; сумма, статус).
- **AttendanceLog / cron** — источник для расчёта зарплат учителей (по проведённым занятиям) и уведомлений о пропусках.
- **Conversation / Message** — чаты омниканального инбокса (канал, контакт, сообщения, статус).
- **Broadcast** — SMS-рассылка (тема/текст, аудитория: все студенты / все учителя / оба, дата, автор, статус доставки).
- **Notification** — журнал отправленных уведомлений (Telegram/SMS).

> ✅ Система оценивания: **5-балльная** (2–5). Оплата — помесячная **от даты вступления студента**. Ставка урока для зарплаты учителя — **гибкая** (задаётся per-занятие/группа). Типы курсов — **настраиваемый справочник**.

## 6. Интеграции

### 6.1. Telegram-боты (студенты и учителя)

- Каждый студент и учитель **привязывает Telegram** к своему аккаунту (через код/deep-link).
- Через бота доступны: **успеваемость, оценки, расписание, пропуски** (для учителя — его группы/расписание).
- Реализация: Telegram Bot API, связка `telegram_chat_id` ↔ `User`.

### 6.2. WhatsApp + Instagram (менеджер продаж)

- Приём входящих сообщений из обоих каналов в единый инбокс CRM.
- Ответ из CRM с доставкой в исходный канал.
- ✅ **Решено: официальные API Meta** — WhatsApp Business Cloud API + Instagram Messaging API (Graph API).
  - Потребуется: аккаунт Meta Business, верификация бизнеса, подтверждённый номер/бизнес-аккаунт Instagram, webhook для входящих сообщений, обработка 24-часового окна сообщений (шаблоны для WhatsApp вне окна).
  - Учитываем затраты Meta на сообщения и сроки на верификацию.

### 6.3. SMS родителям

- SMS-шлюз для отправки уведомлений на номера родителей.
- ✅ **Провайдер: payom.tj** (таджикский SMS-шлюз). Отправка через его API.
- Реализуем через **абстракцию SmsProvider**, чтобы при необходимости сменить шлюз без переписывания логики.

## 7. Автоматизация и уведомления

### 7.1. Уведомление о пропусках (ключевой сценарий)

- Если студент пропускает **3 занятия подряд** (по расписанию группы), родителям автоматически
  отправляется **SMS** (через payom.tj): «Ваш сын/дочь отсутствует на занятиях уже 3 занятия подряд».
- ✅ **Критерий: 3 пропущенных занятия подряд по расписанию группы** (не календарные дни).
- Реализация: фоновая задача (cron/очередь BullMQ), анализирующая последние занятия каждого студента;
  дедупликация — не слать повторно, пока студент не появится хотя бы раз.
- Текст SMS локализуется (язык родителя/центра).

### 7.2. Telegram-оповещения

- Оценки, изменения расписания, замечания могут дублироваться в Telegram-бот (объём — уточняется).

### 7.3. SMS-рассылка (массовые уведомления)

- Ручная рассылка SMS: **новые курсы, поздравления с днём рождения, объявления** и т.п.
- **Аудитория:** всем студентам, всем учителям или обеим группам сразу.
- **Доступ:** только **основатель и администратор**.
- Отправка через тот же шлюз **payom.tj** (абстракция SmsProvider), очередь BullMQ для массовой отправки без блокировки UI.
- Опционально: рассылка с фильтром по филиалу/курсу (на этапе реализации).
- (Возможна автоматизация поздравлений с ДР по дате рождения — как отдельный сценарий cron.)

## 8. Нефункциональные требования

### 8.1. Производительность и «мгновенность»

- **Оптимистические обновления (Optimistic UI):** все мутации (оценки, посещаемость, чаты и т.д.)
  отражаются в интерфейсе мгновенно, до подтверждения сервера, с откатом при ошибке.
  Реализация — TanStack Query `onMutate` / optimistic updates.
- Real-time обновления чатов и уведомлений — через WebSocket.
- Кэширование (Redis + клиентский кэш), пагинация, индексы БД, отсутствие N+1 запросов.
- Цель: отклик интерфейса < 100 мс на действие, отсутствие «ожиданий» и ручных перезагрузок страницы.

### 8.2. Безопасность

> Формулировка «100% защита» переводится в набор конкретных мер (абсолютной защиты не существует, но закрываем стандартные векторы):

- Аутентификация JWT (access + refresh), безопасное хранение токенов.
- **RBAC** — строгая проверка прав на бэкенде для каждого эндпоинта (не только на фронте).
- Хэширование паролей (bcrypt/argon2).
- Валидация и санитизация всех входных данных (class-validator/DTO).
- Защита от SQL-инъекций (ORM/параметризованные запросы), XSS, CSRF.
- Rate limiting, защита от брутфорса, HTTPS everywhere.
- Шифрование чувствительных данных, разграничение доступа к финансам и персональным данным.
- Логирование действий (аудит), особенно в финансах.

### 8.3. Мультиязычность (i18n)

- 3 языка: **таджикский, русский, английский**, переключение на лету.
- Все тексты интерфейса, уведомлений (SMS/Telegram) локализуются.
- ✅ **Язык по умолчанию — русский.** Валюта финансов — **сомони (TJS)**.

### 8.4. Адаптивность

- Полный адаптив под все устройства: мобильные, планшеты, десктопы.
- **Особое требование:** корректное отображение на мобильных **от 320px до 425px** — без багов, сдвигов и «косяков», идентичное поведение во всём диапазоне.
- Mobile-first подход, тестирование на ключевых брейкпоинтах.

### 8.5. Брендинг и тема оформления

- У центра есть **логотип**. Фирменные цвета — **белый + фиолетовый**.
- Дизайн-система на нашей стороне: чистый современный UI на базе фиолетово-белой палитры,
  логотип используется в шапке/сайдбаре, экране входа, Telegram-уведомлениях.
- **Поддержка Dark Mode:** переключение светлая/тёмная тема, фиолетовый акцент сохраняется в обеих.
  Тема через CSS-переменные/Tailwind `dark:`, выбор пользователя сохраняется (localStorage + учёт системной темы).
- **Сеть из 3 филиалов.** На текущем этапе все пользователи видят все филиалы; филиал — метка на данных для фильтрации и аналитики (в т.ч. финансовой по каждому филиалу и сводной).

## 9. Архитектура (предварительно)

- **Монорепозиторий** или два пакета: `frontend` (React+TS) и `backend` (NestJS).
- Backend: модульная структура NestJS (auth, users, branches, students, teachers, courses,
  groups, schedule, journal, finance, chats, notifications, broadcasts, integrations).
- REST API + WebSocket-gateway для real-time.
- Фоновые задачи: очередь (BullMQ + Redis) для рассылок и cron-проверки посещаемости.
- Единый слой прав (guards + RBAC) на бэкенде.
